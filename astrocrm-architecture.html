Core Principles
Non-negotiable architectural rules
  • 1
    Multi-tenancy first. Every database table has tenant_id. Enforced at two independent layers: application query filter AND PostgreSQL Row Level Security (RLS) via Supabase. One missed filter in code cannot leak data — the database rejects it.
  • 2
    One source of truth. All API calls go through static/modules/core/api.js. Never call fetch() directly from feature modules.
  • 3
    Separation of concerns. Routers handle HTTP. Services handle business logic. Models handle data. Never mix them.
  • 4
    Background tasks for anything slow. Sending email, webhooks, notifications — always async via task queue. Never block an HTTP response.
  • 5
    Feature flags per plan. Every premium feature checks tenant.plan before executing. Never hard-code plan logic in feature code.
  • 6
    Mobile-first CSS. All target users are practitioners on phones — no laptops. Write CSS starting from the smallest screen, scale up with min-width media queries. Never design desktop-first and retrofit mobile later. This is a Phase 1 requirement, not Phase 4.
  • 7
    No secrets in code. All credentials in .env. Never commit .env.
  • 8
    All user-visible strings through i18n. No hard-coded EN/RU/NL/DE/ES text in modules. Every string goes through i18n.js.

Multi-Tenancy Pattern

-- Every table has tenant_id
CREATE TABLE contacts (
  id        UUID PRIMARY KEY,
  tenant_id UUID NOT NULL,  -- always
  name      TEXT,
  ...
);

-- Every query filters by tenant
db.query(Contact)
  .filter(Contact.tenant_id == tenant_id)
  .all()

-- Never trust client to send tenant_id
-- Always extract from JWT

What NOT To Do
Common mistakes that will break the architecture
  • Add features to a single monolithic file — always create a module
  • Query the database without filtering by tenant_id — this leaks data between users
  • Store credentials or API keys anywhere except .env
  • Use SQLite in any environment — use Supabase PostgreSQL for all three environments (dev, staging, prod)
  • Send emails or fire webhooks synchronously in an HTTP handler — use ARQ queue
  • Hard-code plan names or feature lists — use PLAN_FEATURES dict
  • Build auth checks per route — use get_current_tenant dependency
  • Call fetch() directly from feature modules — always use core/api.js
  • Hard-code EN/RU/NL text in modules — always use i18n.js
  • Store JWT tokens in localStorage — keep in memory, refresh via httpOnly cookie
  • Accept payment webhooks without verifying the provider signature
  • Deploy to production without staging passing first
  • Create API endpoints without the /api/v1/ prefix

Code Style Rules
Enforced by ruff (Python) and ESLint (JS) — zero warnings policy

Python (Backend)

  • Type hints on all function signatures
  • Docstring on all service functions
  • All DB-touching code in services/, never in routers/
  • API returns { data: ... } or { error: "..." }
  • All routes under /api/v1/

JavaScript (Frontend)

  • ES modules (import/export), no var, const by default
  • No framework — vanilla JS only (Vite bundler in Phase 2)
  • app.js entry point max ~50 lines
  • Global state only in core/state.js — no module-local copies

Authentication & Sessions

ThreatProtectionHow
Stolen passwordbcrypt hashingPasswords never stored in plain text. bcrypt is slow by design — brute-force takes years.
Brute-force loginRate limitingslowapi: 5 attempts/IP/minute. Account locked for 15 min after 10 failures.
Stolen session tokenIn-memory JWTAccess token stored in JS memory only — not localStorage, not cookies. Stolen only if attacker runs code in your browser.
Persistent session after logoutShort-lived tokensAccess token expires in 15 min. Refresh token (httpOnly cookie) expires in 30 days. Logout invalidates refresh token server-side.
Account takeoverEmail verificationNew accounts require email verification. Password reset only via verified email link.
Weak passwordsPassword policyMinimum 8 characters, at least 1 number. Enforced on both frontend and backend.

API & Frontend Security

ThreatProtectionHow
SQL injectionSQLAlchemy ORMAll DB queries use parameterized statements. Raw SQL is never written. ORM prevents injection by construction.
XSS (cross-site scripting)CSP headers + no innerHTMLContent-Security-Policy header blocks inline scripts and untrusted sources. Frontend never uses innerHTML with user data.
CSRF (cross-site request forgery)SameSite cookies + origin checkRefresh token cookie has SameSite=Strict. API checks Origin header on state-changing requests.
Unauthorised API accessJWT on every routeEvery endpoint requires valid JWT via get_current_tenant dependency. No exceptions.
Cross-tenant data accesstenant_id enforcementEvery query filtered by tenant_id extracted from JWT — never from request body. Tested by automated isolation tests.
Scraping / abuseRate limitingAll public endpoints rate-limited. Auth endpoints stricter (R3).
ClickjackingX-Frame-Options headerX-Frame-Options: DENY — prevents AstroCRM being embedded in a malicious iframe.
MIME type sniffingX-Content-Type-OptionsX-Content-Type-Options: nosniff — browser won't guess content type.

Payment Security

ConcernApproach
Card data storageWe never store card numbers. Stripe/Mollie handle all card processing and are PCI DSS Level 1 certified. AstroCRM only stores a customer_id reference.
Fake webhook attacksEvery inbound webhook verified with provider's HMAC signature before any action is taken. Reject unsigned requests with 401.
Duplicate payment processingWebhooks are idempotent — processing the same event twice has no effect. Stripe/Mollie event IDs stored to detect duplicates.
Russian payment providers (YooKassa, Prodamus)Same webhook signature verification. YooKassa uses SHA-1 HMAC. Prodamus provides its own signing method. Both verified before processing.

Known Risks & Mitigations
Critical review — April 2026. Every risk has a fix and a deadline.

🔴 Critical — Security & Data Loss

#RiskWhy it mattersFixWhen
R1 No multi-tenancy tests One forgotten tenant_id filter = user sees strangers' data pytest: create 2 tenants, call API as tenant A, assert tenant B data never appears Phase 1
R2 JWT token in localStorage Any XSS vulnerability in any script = all users' tokens stolen Store token in memory only. Silent refresh via httpOnly cookie on page load Phase 1
R3 No rate limiting on auth Login endpoint can be brute-forced — 100K password attempts in minutes slowapi — 5 login attempts/IP/min, 3 password resets/hour Phase 1
R4 Background tasks not persistent Server restart during deploy = queued emails lost. Paid user gets no confirmation. Replace FastAPI BackgroundTasks with ARQ (Redis-backed queue). Jobs survive restarts. Phase 1
R5 Webhook signatures not verified Anyone can POST to payment webhook and upgrade their account for free Verify provider signature before acting on any webhook. One function per provider. Phase 2
R6 SQLite dev ≠ PostgreSQL prod Different handling of dates, JSON, LIKE, case sensitivity. Bugs found only in production. Run PostgreSQL locally via Docker: docker run -p 5432:5432 postgres Phase 1

🟠 High — Operational Gaps

#RiskWhy it mattersFixWhen
R7 No error tracking Broken features go undetected. Most users don't complain — they just leave. Sentry free tier — one line in main.py. Full stack trace + email on new errors. Phase 1
R8 No structured logging Can't debug production — no record of what happened, which user, which endpoint. structlog — every request logs tenant_id, endpoint, duration, status. Ship to Papertrail (free). Phase 1
R9 No GDPR export / delete Legal requirement. Every user can demand data export or full deletion. Non-compliant before first NL client. GET /api/v1/tenant/export (ZIP download) and DELETE /api/v1/tenant/account. Self-serve. Pre-launch
R10 No API versioning Changing any response field breaks existing clients. Impossible to fix after mobile app ships. All routes under /api/v1/ now. Breaking changes go to /api/v2/. Zero cost to add now. Phase 1
R11 No /health endpoint Load balancer can't check if app is alive. Dead instance keeps receiving traffic. GET /health returns {"status":"ok","db":"ok"} — also pings DB. Phase 1

🟡 Medium — Developer Handover

#RiskWhy it mattersFixWhen
R12 No README / setup guide No developer can run the project without reading all code first. README.md — prerequisites, .env setup, run locally, run tests, deploy to staging. Phase 1
R13 No linter / formatter Code style drifts across sessions. Developer can't distinguish bugs from style choices. ruff for Python, ESLint for JS. Both run in CI before every deploy. Zero warnings policy. Phase 1
R14 No tests at all No developer can change anything safely. Every modification might silently break something. pytest for backend (auth, isolation, webhooks). Vitest for JS. Target 80% on services/. Phase 1
R15 models.py as single file At AmoCRM scope = 1,500+ lines. No developer can navigate it. Split: models/contacts.py, models/deals.py, models/auth.py etc. with __init__.py. Phase 2
R16 No bundler (Vite) 15+ modules = 15+ browser requests per load. No minification. Performance degrades at scale. Add Vite in Phase 2 — wraps existing ES modules with zero structural changes. 2 hours. Phase 2

Database Design Standards
Indexes · Constraints · Soft delete · Zero-downtime migrations

Required indexes — every new table

-- Add to every Alembic migration
CREATE INDEX idx_contacts_tenant
  ON contacts(tenant_id);
CREATE INDEX idx_deals_tenant
  ON deals(tenant_id);
CREATE INDEX idx_contacts_email
  ON contacts(tenant_id, email);
CREATE INDEX idx_contacts_created
  ON contacts(tenant_id, created_at DESC);
-- Rule: index every WHERE, ORDER BY, JOIN column

Required columns — every model

tenant_id  = Column(UUID,
  ForeignKey("tenants.id"),
  nullable=False, index=True)
created_at = Column(DateTime,
  default=func.now(), nullable=False)
updated_at = Column(DateTime,
  default=func.now(),
  onupdate=func.now(), nullable=False)
# Soft delete — NEVER hard-delete user data
deleted_at = Column(DateTime, nullable=True)
# All queries add:
# .filter(Model.deleted_at == None)
Zero-downtime migration pattern (required for live tables):
1. Expand — add new column (nullable), deploy  →  2. Migrate — backfill data in background, deploy  →  3. Contract — remove old column, deploy
Never rename or drop a column in a single step on a live table. This locks the table and causes downtime.

             mirrors production

hotfix/xxx  — urgent prod fixes only

Rule: nothing merges to main without
a pull request. Claude reviews all
developer PRs (and vice versa).

Deployment process

  1. Feature branch → PR → reviewed → merged to main
  2. GitHub Actions auto-deploys main to staging
  3. Manual check on staging.astrocrm.app
  4. One-click promote to production via GitHub Actions

Accessibility (a11y)
WCAG 2.1 Level AA — required from Phase 1
  • ✓ All buttons/inputs have aria-label if no visible text label
  • ✓ Colour contrast ratio ≥ 4.5:1 for text, ≥ 3:1 for large text
  • ✓ All interactive elements reachable by keyboard (Tab key)
  • ✓ Focus visible on all elements — never outline: none
  • ✓ Error messages linked to inputs via aria-describedby
  • ✓ Icon-buttons have alt text
Why it matters: Practitioners with visual impairments are potential customers.

Legal risk: The European Accessibility Act (June 2025) tightens requirements in EU markets. WCAG 2.1 AA compliance reduces legal exposure in NL and DE.

Mobile benefit: Accessible UI is also better mobile UX — larger tap targets, clear labels, logical focus order.

Activity & Interaction Timeline
The core CRM entity — without this, you have a contacts list, not a CRM
Why this matters: Contacts, deals, and pipelines describe state. Activities describe what happened — the history of every interaction. This is the foundation of automations, analytics, and any future AI features. Missing from Phase 1 = building a CRM without the CRM core.

Activity data model

class ActivityType(Enum):
  note        = "note"      # Session notes
  email_sent  = "email_sent"
  email_recv  = "email_recv"
  call        = "call"
  meeting     = "meeting"   # Appointment
  task        = "task"      # To-do item
  deal_moved  = "deal_moved"# Auto on stage change
  automation  = "automation"# Auto-triggered

class Activity(Base):
  id         UUID PK
  tenant_id  UUID FK (indexed)
  contact_id UUID FK nullable
  deal_id    UUID FK nullable
  type       ActivityType
  content    Text  # note text, subject…
  metadata   JSONB # flexible per type
  created_by UUID FK
  created_at DateTime
  # No updated_at — append-only

API endpoints (Phase 1)

GET /api/v1/contacts/{id}/activities

What plugs into Activity

  • 📝 Session notes → type=note, contact_id=…
  • 🔄 Pipeline moves → type=deal_moved auto-created

  • ⚡ Automations → type=automation logged
  • 📊 Reports → query by type + date range
  • 🤖 AI (Phase 4) → timestamped record of everything

Multi-Tenancy Enforcement — Supabase Row Level Security
DB-level enforcement, not just application-level — one forgotten filter can't leak data
The problem with query-only enforcement: If every developer must remember to add .filter(tenant_id == ...) to every query, one forgotten filter = data from another user's account is exposed. Discipline is not enough at scale. The database must enforce it independently.

PostgreSQL RLS policy (Supabase)

-- Enable on every table
ALTER TABLE contacts
  ENABLE ROW LEVEL SECURITY;

-- Policy: DB rejects rows from
-- other tenants automatically
CREATE POLICY tenant_isolation
  ON contacts
  USING (
    tenant_id =
    current_setting(
      'app.current_tenant'
    )::uuid
  );

-- In FastAPI: set context per request
async def get_db_session(tenant_id):
  db.execute(
    "SET app.current_tenant = :tid",
    {"tid": str(tenant_id)}
  )
  return db

Why this matters

  • 🔒 Query without tenant filter → DB returns zero rows, never wrong tenant's data
  • 🔒 Developer forgets filter → still safe
  • 🔒 Works at DB level, independently of application code
  • ✅ Supabase has native RLS support — built-in, no extra cost
  • ✅ Stacks on top of the application-level filter — two independent layers
Optimistic locking — concurrent edits
When two sessions update the same record simultaneously, last write silently wins. Fix: compare updated_at before writing. If it doesn't match, return 409 Conflict — "Reload and try again." Frontend sends the updated_at it last saw. One check per write, zero lost data.

Idempotency — webhook safety

async def process_stripe_webhook(
    ctx, event_id: str, data: dict):

  # Already processed? Skip safely
  if await db.get(ProcessedWebhook,
      event_id):
    return  # not an error

  await handle_payment_success(data)

  # Mark processed (one-time only)
  db.add(ProcessedWebhook(
    id=event_id,
    processed_at=datetime.utcnow()
  ))
  await db.commit()

After max_tries failures, the job is marked failed in Redis. Sentry captures the exception on each retry. Review failed jobs weekly — a silent failure here means a user never received their welcome email or their payment wasn't processed. Never let a failed job disappear without a trace.
