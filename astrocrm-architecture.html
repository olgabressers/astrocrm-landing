Core Principles
Non-negotiable architectural rules
  • 1
    Multi-tenancy first. Every database table has tenant_id. No data ever leaks between users. Enforce at query level, not just UI.
  • 2
    One source of truth. All API calls go through static/modules/core/api.js. Never call fetch() directly from feature modules.
  • 3
    Separation of concerns. Routers handle HTTP. Services handle business logic. Models handle data. Never mix them.
  • 4
    Background tasks for anything slow. Sending email, webhooks, notifications — always async via task queue. Never block an HTTP response.
  • 5
    Feature flags per plan. Every premium feature checks tenant.plan before executing. Never hard-code plan logic in feature code.
  • 6
    Mobile-first CSS. All target users are practitioners on phones — no laptops. Write CSS starting from the smallest screen, scale up with min-width media queries. Never design desktop-first and retrofit mobile later. This is a Phase 1 requirement, not Phase 4.
  • 7
    No secrets in code. All credentials in .env. Never commit .env.
  • 8
    All user-visible strings through i18n. No hard-coded EN/RU/NL/DE/ES text in modules. Every string goes through i18n.js.

Database & Authentication
SQLite (dev) → PostgreSQL (production) · JWT tokens

Multi-Tenancy Pattern

-- Every table has tenant_id
CREATE TABLE contacts (
  id        UUID PRIMARY KEY,
  tenant_id UUID NOT NULL,  -- always
  name      TEXT,
  ...
);

-- Every query filters by tenant
db.query(Contact)
  .filter(Contact.tenant_id == tenant_id)
  .all()

-- Never trust client to send tenant_id
-- Always extract from JWT

Authentication Flow

  • 🔐 JWT access token — short-lived (1 hour)
  • 🍪 Refresh token — httpOnly cookie (30 days)
  • 🔑 Password hashing: bcrypt
  • 📧 Email verification on registration
  • 🔄 Password reset via email token

Database

  • 💻 Dev: SQLite (local, single developer)
  • 🚀 Prod: PostgreSQL (Supabase free tier)
  • 🔧 ORM: SQLAlchemy
  • 🔄 Migrations: Alembic

PLAN_FEATURES = {
  "standard": [
    "pipeline", "contacts",
    "campaigns_basic", "reports"
  ],
  "full": [
    "pipeline", "contacts",
    "campaigns_advanced", "reports",
    "two_businesses", "automations",
    "telegram", "whatsapp"
  ],
  "enterprise": ["*"]  # all features
}

def require_feature(feature: str):
  def dependency(tenant = Depends(...)):
    if feature not in PLAN_FEATURES
        .get(tenant.plan, []):
      raise HTTPException(
        403, "Upgrade your plan")
  return Depends(dependency)

Development Phases
Foundation must be solid before building features

Phase 1 — Foundation

  • Refactor app.js into modules
  • Refactor app.py into routers + services
  • JWT authentication (register, login)
  • Multi-tenancy (tenant_id all tables)
  • Switch to PostgreSQL (Supabase)
  • Deploy to Railway/Render
  • Mobile-first CSS ★

Phase 2 — Integrations

  • Gmail OAuth connect
  • SMTP (Yandex, Mail.ru)
  • Pipeline automation emails
  • Campaign sends via connected email
  • Telegram lead capture bot
  • Stripe + Mollie payments

Phase 3 — Monetisation

  • Subscription plans in database
  • Feature gating middleware
  • Plan upgrade/downgrade flow
  • WhatsApp integration
  • Advanced automations

Phase 4 — Growth

  • Advanced analytics
  • Third-party API
  • Admin dashboard (all tenants)
  • Marketplace / integrations hub
Even though the final product targets AmoCRM-level complexity, Phase 1 still makes complete sense — in fact it's more important than ever. You cannot safely add 60+ features on top of a single-file app. The foundation (auth + multi-tenancy + module structure) is what makes every future feature possible without rewriting.

What NOT To Do
Common mistakes that will break the architecture
  • Add features to the single-file app.js — always create a module
  • Query the database without filtering by tenant_id
  • Store credentials or API keys anywhere except .env
  • Use SQLite in production
  • Send emails synchronously inside an HTTP request handler
  • Hard-code plan names or feature lists in feature code — use PLAN_FEATURES
  • Build authentication from scratch for each route — use the dependency
  • Call fetch() directly from feature modules — always use core/api.js
  • Hard-code EN/RU/NL text in modules — always use i18n.js

Code Style Rules
For Claude and any developer working on AstroCRM

Python (Backend)

  • Type hints on all function signatures
  • Docstring on all service functions
  • All DB-touching code in services/, never in routers/
  • API returns { data: ... } or { error: "..." }

JavaScript (Frontend)

  • ES modules (import/export), no var, const by default
  • No framework — vanilla JS only
  • app.js entry point max ~50 lines
  • Global state only in core/state.js — no module-local copies
